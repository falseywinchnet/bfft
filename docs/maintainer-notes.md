# Maintainer notes for GitHub repository settings

These changes cannot be applied from inside the working tree. Configure them in
the GitHub UI while preparing the beta and public release.

## Repository appearance

- Add a repository description: `Bruun real FFT library for C and C++`.
- Add topics: `fft`, `dsp`, `c`, `cpp`, `simd`, `bruun-transform`.
- Set the website field to the hosted documentation URL once available.
- Candidate README graphics live in `docs/assets/`; pick one for the repository
  social preview when desired.

## CI and branch protection

A first workflow is tracked at `.github/workflows/ci.yml`. After the first green
hosted run:

- Protect the default branch.
- Require pull requests before merging.
- Require these status checks before merging:
  - `make (ubuntu-latest)`
  - `cmake (ubuntu-latest)`
- Require branches to be up to date before merging once contributor volume makes
  that useful.
- See `TODO_HUMAN.md` for a step-by-step GitHub Actions and branch-protection
  manual.

## Release assets

- Publish source archives generated by GitHub releases.
- Attach prebuilt packages only after platform-specific CI exists.
- Include the MIT license in all release artifacts.
- Pin or highlight `examples/benchmark.cpp` in release notes if benchmark claims
  are included.

## Future CI expansion

The first required CI is intentionally small and Linux-only. Future matrix work
should add:

- Linux GCC and Clang variants.
- macOS Clang after Makefile shared-library naming and C API link details are
  validated there.
- Windows MSVC and MinGW after DLL export/import policy is designed.
- x86_64 AVX2-capable and scalar fallback builds.
- aarch64 NEON runners when hosted runner support is practical.
