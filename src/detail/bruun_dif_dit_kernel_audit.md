## 1. The two walks, stated once, in phase space

Both kernels compute the same object: a factorization of the real DFT
`F : R^N -> R^N` (r2c pairing) into `F = ‚àöN ¬∑ Q` with `Q ‚àà O(N)`. Every
primitive in the normalized ("unit-frame") basis is one of exactly two maps:

1. **The Givens step** (`norm_q_fwd` / `pair_reduce{_v2,_v4f}_norm`):
   with `r = c¬∑oa ‚àí s¬∑ob`, `i = s¬∑oa + c¬∑ob` (a rotation of the odd child by a
   point `(c, s) ‚àà S¬π`),

   ```
   (ea, eb, oa, ob)  ->  (ea + r, eb + i, ea ‚àí r, i ‚àí eb)
   ```

   Direct computation gives `‚Äñout‚Äñ¬≤ = 2‚Äñin‚Äñ¬≤` **exactly** (the rotation
   preserves `oa¬≤ + ob¬≤`; the ¬± combine doubles energy). So each butterfly is
   `‚àö2 √ó` an orthogonal map of R‚Å¥.

2. **The binomial step** (`binomial_fwd`, seed adds, DC chains):
   `(a, b) -> (a+b, a‚àíb)` = `‚àö2 √ó` a reflection.

Consequence ‚Äî the hypersphere picture the code implicitly lives on:

- The full state vector after `k` levels lies on the hypersphere
  `S^{N‚àí1}` of radius `2^{k/2}¬∑‚Äñx‚Äñ`. The forward walk is a **conformal spiral**:
  a pure rotation of R^N composed with a uniform dilation `‚àö2` per level. The
  inverse distributes the contraction as the `0.5` factors inside
  `pair_expand_norm` / `binomial_inv`, composing to exactly `1/N` ‚Äî this is why
  neither kernel has (or needs) a normalization pass, and why round-trip error
  is machine-epsilon *by construction* whenever forward and inverse read the
  same `(c, s)` tables (see I1 for the sharp edge of that "whenever").
- Every twiddle is a point on the unit circle S¬π; a butterfly never scales
  anisotropically. Numerical energy cannot grow in any single component faster
  than the global `‚àö2` ‚Äî the raw Bruun basis (`c2 = 2cosŒ∏`, dead code in the
  DIT, see I5) violated exactly this, letting the twiddle magnitude reach 2 and
  paying `1 ‚àí c2¬≤` cancellations (~1e‚àí12 round trip at N=2¬≤‚Å∞ per the header
  notes). The normalized basis is the hypersphere-aligned formulation.
- Both kernels use the **same leaf frame**: a leaf residue pair is directly
  `(Re X_k, ‚àíIm X_k)`. DIF: `residues_to_complex` (`X[k].re = v[2m]`,
  `X[k].im = ‚àív[2m+1]`); DIT: the `_norm` terminals' sign flip. The two
  kernels' spectra are therefore frame-compatible at the leaves; only the
  *ordering* differs.

### 1.1 The angle sets (where the twiddles sit on S¬π)

- **DIF** (`init`, `bruun_dif_kernel.hpp:1571-1605`): angles are generated by
  the half-angle covering map
  `Œ±(1) = œÄ/4, Œ±(2m) = Œ±(m)/2, Œ±(2m+1) = œÄ/2 ‚àí Œ±(m)/2` ‚Äî the inverse of the
  angle-doubling map on the quarter circle, folded dihedrally. Leaves land on
  the dyadic odd multiples `œÄ¬∑(2j+1)/N`. Two elegant consequences the DIT does
  not share:
  - `C[]` stores only cosines; the sine of node `m` is the cosine of its
    sibling: `s_twiddle(m) = C[m ^ 1]` (complementary angles). One table,
    half the storage, and the *stable* half-angle recurrence
    `ce = ‚àö(¬Ω(1+c))`, `se = s/(2¬∑ce)` avoids the `‚àö(1‚àíc¬≤)` cancellation ‚Äî the
    init comment at `bruun_dif_kernel.hpp:1593-1596` explicitly warns about it.
  - Plan setup needs O(1) libm calls (one `sqrt(0.5)` seed); everything else is
    sqrt/divide recurrences.
- **DIT** (`reset`, `bruun_dit_kernel.hpp:152-297`): angles are sampled
  directly, `Œ∏ = 2œÄk/s` per merge stage, one `cos`/`sin` libm call per table
  entry across seven-plus tables. The per-*position* angles are the defining
  difference from the DIF (see ¬ß1.2), so a recurrence cannot collapse them ‚Äî
  but the *construction* of `s` commits the exact cancellation the DIF's
  comment warns about (I1, measured).

### 1.2 The structural duality (why the two walks are shaped differently)

|                                | DIF (factor-tree descent)                | DIT (spectral ascent)                       |
|--------------------------------|------------------------------------------|---------------------------------------------|
| Twiddle variation              | per **node**, uniform across the block   | per **position**, uniform across blocks     |
| What that forces on SIMD       | nothing ‚Äî vertical SIMD is free at any width | mirror-lane packing; one crossing zone   |
| Lane-crossing location         | leaf codelets only (in-register 4√ó4 transpose) | terminal merge only (fused with projection) |
| Twiddle bandwidth              | O(#nodes) scalars                        | O(N) per top stage (streamed, shared across blocks) |
| Output permutation             | fused scatter (heap-opt order) or two-phase pack | none ‚Äî standard order falls out of the walk |
| Input permutation              | none (natural order in)                  | bit-reversed comb gather at the seed        |

One angle applied to a whole block is an **isoclinic rotation** of `R^{2q}`
(q planes rotated by the same angle simultaneously) ‚Äî isoclinic rotations
vectorize at *any* register width with zero shuffles, which is why the DIF
gained an AVX2 path by mechanical widening. The DIT's per-position angles
break isoclinicity across adjacent slots, so each doubling of SIMD width must
*manufacture* a symmetry: a top index bit whose two halves run identical
twiddles. That is the **pairing-axis law** of this codebase:

- 2 √ó f64: mirror bit `b ‚Üî b + N/4` (rev-adjacent seeds; crossing = last merge).
- 4 √ó f32: mirror bit + `cA/cB` bit `s ‚Üî s + N/8` (crossing = last two merges).
- 4 √ó f64 (future AVX2, per `notes/experimental_radix4_dit_avx2_handoff.md`):
  the f32 pairing scheme verbatim with f64 lanes ‚Äî each width doubling
  consumes one more top bit and deepens the terminal crossing zone by exactly
  one merge. The interior stays byte-identical vertical SIMD. This is worth
  writing down because it is the *entire* design of any future `bruun_v4`
  port: no new math, one more `CATLO/CATHI/ZIPLO` stage in the terminal.

The interpolation inverse (`itp_*`, currently unrouted ‚Äî I4) is the exact dual
cell of this table: CRT interpolation up the factor tree has per-**block**
constants, so its interior is free vertical SIMD at any width and its single
shuffle zone is the spectral *head*. The codebase thus contains all four
corners of the (decimation √ó direction) square; only two are routed.

### 1.3 The walk order (transport, not arithmetic)

- **DIF forward** (`FWD_SCHEDULE`, built at `bruun_dif_kernel.hpp:2684`): for
  each halving segment `[h, 2h)` from `h = N/2` down to 32: depth-first
  two-level-fused descent (`norm2_fused` ‚Üí quarters) bottoming in the
  depth-3 leaf codelets, then one `binomial_fwd` on `[0, h)`. Segments are
  finished completely before the walk moves on ‚Äî the working set halves at
  every step, so the top of the tree is streamed once and the bottom is
  cache-resident. Leaves either scatter straight to `X` (fused scatter, native
  heap-opt order chosen by the DP orientation optimizer at init, which
  minimizes max cyclic bin travel so the scatter is piecewise-local) or stay
  in residue order for the two-phase standard pack.
- **DIT forward** (`compute_vwork_norm`, `bruun_dit_kernel.hpp:1877`): seed
  blocks in bit-reversed *block* order (turns the bit-reversed gather into
  ~2¬∑SB sequential comb streams), each block fused through all merges that fit
  (`s_target`), then plain per-stage radix-4 sweeps over the whole array for
  the remaining top stages, then the terminal crossing fused with projection.
  The inverse mirrors it, with the row-tile scatter
  (`seed_scatter_row_tiles_v`) keeping the *store* side cache-local for large
  N (`nb >= 64`).

The asymmetry to hold onto: above the fused-block size, **DIF touches each
region O(1) more times (descending segments), DIT touches the whole array once
per radix-4 level-pair**. For N ‚â´ LLC the DIT pays `(log‚ÇÑ(N/s_target))` full
sweeps where the DIF pays a geometrically-decaying series. The measured NEON
ratios in `notes/` (DIT fwd 0.89‚Äì1.00 √ó DIF) say the DIT buys that back with
its zero-permutation output and pure-vertical interior; on machines with
smaller LLCs the balance will tilt toward DIF at large N. Worth keeping both.

## 2. Register-pressure audit (per hot kernel, per ISA, per precision)

Register budget context: x86 SSE2/AVX2 = 16 architectural vector registers;
NEON (aarch64) = 32. At a fixed register *width*, double and float have the
**same live-vector counts** ‚Äî precision changes footprint per lane, not the
DAG. So the pressure findings below are structural; the double/float
differences that exist are load-*scheduling* differences, not width ones.

**Reflection ("why are you loading all 16 before storing anything??"),
`merge4_v_norm`:** the `(i, M‚àíi)` palindromic pairing exists so the in-place
merge is clobber-free: `oi`'s stores hit slots `{i, 2M‚àíi, 2M+i, 4M‚àíi}` while
`om`'s loads need `{M‚àíi, 2M‚àíi, 3M‚àíi, 4M‚àíi}` ‚Äî the intersection
`{2M‚àíi, 4M‚àíi}` forces `om`'s loads before `oi`'s stores, hence 16 concurrent
loads and the spill pressure. *Resolution:* the hazard is only 2 of `om`'s 4
node reads. A legal reordering exists:

1. load `i`-inputs (8), compute `oi` (inputs die ‚Üí 8 live);
2. load only `om`'s hazard slots `{2M‚àíi, 4M‚àíi}` (4 vectors ‚Üí 12 live);
3. store all of `oi` (‚Üí 4 live);
4. load `om`'s remaining slots `{M‚àíi, 3M‚àíi}` (4 ‚Üí 8 live), compute `om`, store.

Peak drops from ~26‚Äì30 to ~14‚Äì16 vectors ‚Äî inside even the 16-register
budget. Same restructuring applies to `split4_v_norm`, `merge4_v4f`,
`split4_v4f`. This is the highest-leverage register-pressure change in either
file **for x86-128 targets**; on NEON the win is smaller (spills there are
few) and the benchmark gate decides. It changes no math and no memory
ordering semantics (the hazard set is honored explicitly instead of
wholesale).

### 2.3 Double vs float, summarized

- At fixed width, float does **not** relieve register pressure ‚Äî same vector
  counts. It relieves *cache* pressure (¬ß3) and doubles arithmetic per vector.
- The only real double/float register divergence is deliberate load
  scheduling: DIF f32 kernels are B-first, DIF double kernels are not
  (gated); DIT kernels are width-uniform.
- The DIT f32 *terminals* are the one place float is structurally worse:
  `terminal_v4f_f32` (:2589) computes lanes {2,3} as duplicates of {0,1}
  (50 % of terminal vector work wasted) and exits through stack spill-outs
  (`tfa[4]` stores + scalar re-loads); `terminal_inv_v4f_f32` (:3587) gathers
  through 4√ó`V4F_SET4` = up to 16 scalar inserts per iteration. The interior
  is immaculate; the terminals are the f32 walk's rough edge. *Proposal:*
  process two `i` values per terminal iteration so all 4 lanes carry payload,
  and store via `ZIPLO`/`CATLO` pairs to `X[i], X[i+1]` the way
  `terminal_radix2_v_norm_f32` (:2561) already does. O(N/8) iterations, so
  this matters only in the small-N regime where the terminal is a visible
  fraction ‚Äî measure first.

## 3. Cache / eviction-pressure audit

### 3.1 Footprints (work arrays only)

| N | DIF/DIT double work (8N B) | f32 work (4N B) | vs typical L1d (32‚Äì128 K) | vs typical L2 (512 K‚Äì16 M) |
|---|---|---|---|---|
| 4096 | 32 K | 16 K | L1-resident | ‚Äî |
| 16384 | 128 K | 64 K | edge of L1 | L2-resident |
| 65536 | 512 K | 256 K | ‚Äî | L2-resident |
| 1M | 8 M | 4 M | ‚Äî | edge of / beyond L2 |

The blocking decisions in both kernels are keyed on **element counts**
(`n >= 4096`, `s_target = 256`, `nb >= 64`, two-phase at `N >= 8192`,
`BRUUN_ITP_TILE_* / TIER2` in doubles). A float array of the same element
count is *half the bytes*, so every one of these thresholds fires at half the
byte-footprint for f32 ‚Äî i.e., f32 starts paying blocking overhead while
still comfortably cache-resident, or (where thresholds were hand-adjusted)
diverges from the double path silently. Evidence this is already biting: the
three sibling drivers use three different comparisons ‚Äî

- `compute_vwork_norm` (double): `n_ >= 4096 ? 256 : chain_limit` (:1881)
- `compute_vwork_v4f` (f32 fwd): `n_ == 4096 ? 256 : chain_limit` (:2332)
- `compute_vwork_inv_v4f` (f32 inv): `n_ > 4096 ? 256 : chain_limit` (:3703)

`>` for f32 is byte-consistent with `>=` for double (f32 at 8192 elements =
double at 4096 bytes-wise). **`==` is neither**: for every f32 size above
4096 the forward seeds one monolithic block spanning the entire array ‚Äî no
cache blocking at all, and the bit-reversed block loop degenerates to a
single iteration. This matches tracked follow-up (c) in
`notes/dit_capability_checklist.md` ("fold the forward's fused-seed/bitrev
block order into compute_vwork_v4f ‚Äî currently a plain schedule") and
explains the checklist's own fwd_x dip at 262144 (0.82 is suspiciously *good*
‚Äî NEON's LLC hides it; on small-L2 parts this will be a cliff).
*Recommendation (P0-perf):* make it `n_ > 4096 ? 256 : chain_limit` to match
the f32 inverse, and ‚Äî systemically ‚Äî express every blocking threshold in
**bytes** (`n * sizeof(T)`) or scale by `sizeof(double)/sizeof(T)` so the twin
paths cannot drift again. One helper, seven call sites.

### 3.2 Metadata streams that compete with data

- **DIF schedule stream:** `FwdOp` is 32 bytes (`base,q,m,kind` +
  `MemHint{base,span,align,stream,next_base}`); a transform executes roughly
  N/6 ops, so the schedule stream is ~5.3N bytes ‚Äî **two-thirds of a full
  data-read pass** spent on metadata, every transform. Of `MemHint`, only
  `next_base` is ever read, and only by the two *inverse* runners' prefetch
  (`:3728`, `:3108`); the forward runners read nothing from it. `align`,
  `stream`, `span` are write-only fields. *Resolution:* shrink `FwdOp` to 16
  bytes (`base:u32, q:u32, m:u32, kind:u16, pad/next_delta:u16` ‚Äî or keep a
  32-bit `next_base` and drop the rest). Halves the metadata bandwidth of
  every scheduled transform; zero math risk. If the MemHint fields were meant
  for a future scheduler, park the struct in a comment or the notes, not in
  the hot stream.
- **DIF `LeafTw`:** actually 96 bytes (32 c4 + 16 c2 + 8 c1 + 8 s1 + 32 idx),
  not the "144-byte entry" the comment at :1756 claims (comment rot from an
  older layout ‚Äî fix the comment). Residue-order transforms
  (`codelet_d3_tw_res*`) stream the same records but never read `idx[8]` ‚Äî
  33 % of that stream is dead weight on the residue/filter paths. Trade-off:
  a split `TW_RES` (64 B) duplicates 6.7N bytes of table; only worth it if
  the residue path is a first-class product (it is, for `filter_signal`).
  Note and defer.
- **DIT twiddle streams:** per-position tables mean the top merge stage reads
  ~s/2 doubles of `cs_` per 2s-double block ‚Äî 25 % table overhead at the top
  levels, amortized across blocks lower down (shared pointers, cache-hot).
  Unavoidable given ¬ß1.2; the mirrored-lane design already minimizes it
  (twiddles broadcast, read once per position pair instead of per node).

### 3.3 Plan-time memory and the oracle tax

DIT `reset()` builds, for every plan: `rev_` (N/2 ints), `cos_`, `neg_sin_`,
`inv_neg_sin_` (3 √ó (N/2+1) doubles), `twiddle_` (~N/2), `inv_twiddle_` (~N),
`cs_` (~N), `term_tw_` (0.75N), `iterm_tw_` (~N), `fterm_cs_` (0.75N), plus
the **entire interpolation-inverse plan** (`ihead_*`, `ir2_tw_`, `isw_tw_`,
`itile_*` ‚âà 1.6N doubles, a std::sort over tiles, and thousands of libm
calls). Total ‚âà 7‚Äì8N doubles of tables. Of these, **the raw-basis walk and
the interpolation inverse are unrouted** (kept "as reference/oracles", per
the AVX2 handoff note) ‚Äî so `inv_twiddle_`, `iterm_tw_`, `inv_neg_sin_`, and
all six `itp` arrays (~3.6N doubles, roughly half the table budget) are
allocated, initialized, and then evict user data from cache **for paths that
cannot be reached**. `term_tw_` is live only as an *input* to building
`fterm_cs_` (:288-289). The DIF, by contrast, carries ‚âà 2.5‚Äì3N doubles-equiv
and every table is reachable.

*Reflection ‚Üí resolution (I4/I5):* keeping oracles is good practice; paying
for them per-plan is not. Gate the raw tables and `build_interp_plan()`
behind a `BRUUN_DIT_BUILD_ORACLES` compile flag (or a lazy build on first
use), and fold the two `term_tw_`-derived constants directly into the
`fterm_cs_` computation. Plan setup drops ~half its memory and most of its
libm time; the oracle code stays compilable and testable.

### 3.4 Stream-count pressure (the subtle one)

- DIT `terminal_v_norm`: 2 read streams (cA, cB) + `fterm_cs_` stream + 4
  write streams into X moving in opposing directions = 7 concurrent streams.
  Fine everywhere.
- `itp_terminal_v`'s comment (:3910) records that a fused 12-stream version
  lost ~20 % at L2 sizes to the 5-stream two-half-pass version ‚Äî direct
  evidence this codebase has already hit hardware prefetcher stream-tracking
  limits. Keep that comment; it is the institutional memory for why nobody
  should "re-fuse" it.
- DIF `norm2_fused` at top levels: 8 strided read+write streams
  (4 quarters √ó 2 offsets). Within prefetcher budgets, but it plus the
  32-byte schedule stream plus `C[]` picks is the likely reason the
  measured-and-gated double B-first stagger was noisy: the kernel is
  store/stream-bound, not ALU-bound, exactly as the flow notes conclude.

### 3.5 Double vs float eviction, summarized

f32 halves every footprint and every stream, so at equal N the f32 walk sits
one full cache tier lower ‚Äî *except* where the element-count thresholds (¬ß3.1)
make it block too early/too late, and except the DIT f32 odd-size terminal,
which round-trips N floats through the output buffer as scratch
(`forward_simd_f32_wide` :2640-2650: depair into `output`, merge in place,
copy back into `work`, then the radix-2 terminal re-reads `work` and writes
`output`). The copy exists because the terminal cannot run in place over its
own input (write range overlaps read range at i‚â•1). It is N floats of pure
transport ‚âà 1/(2¬∑log N) of total traffic ‚Äî acceptable, but worth a comment in
the code; today the copy loop looks like an accident. (Checked: it is not ‚Äî
alias analysis confirms the hazard.)

## Odd-size inverse terminal: fused three-level cell (2026-07-03)

Goal: remove the extra transport that odd `log2(N)` inverse sizes pay relative
to even sizes. Odd sizes end the forward with a radix-2 terminal, so the inverse
runs a radix-2 crossing *then* a full split chain; the even terminal is already
a 2-level (radix-4) cell that writes the post-terminal work layout directly. The
proposal was to make the odd terminal a fused three-level crossing cell
(radix-2 + first radix-4 split) reading standard bins and writing the post-split
layout, saving the intermediate work round trip.

Implemented, measured, and **only the narrow f32 fusion survived**. The results
are a clean illustration that these terminals are *not* transport-bound the way
the framing assumed ‚Äî they are gather/vectorization-bound.

### f64: fused `terminal3_inv_v_norm` ‚Äî REVERTED (measured 8‚Äì21% slower)

Built the exact fused cell (radix-2 crossing per split-input slot, then
`inv_butterfly4_v_norm`, writing the post-split layout; `compute_vwork_inv_norm`
skips the top split level). Roundtrip stayed machine-precision (1.8e-15), so it
was *correct*. But it measured **slower at every odd size** (512: +8%, ‚Ä¶,
524288: +21%). Reason: the even terminal vectorizes its crossing because the
bins it reads come as aligned pairs `{i, 2M-i}` / `{n/2-i, n/4+i}`; the odd
terminal's radix-2 crossing produces the *mirror pair* `{node s, node s+N/4}`,
and the first split's palindromic access reads 4 non-adjacent slots
`{i, 4M-i, 2M-i, 2M+i}`. Fusing therefore forces the crossing to run per-slot
scalar (`pair_expand_cs` + `V2_SETLH` packing) plus a scattered X gather,
instead of the baseline's fully-vectorized 2-adjacent-slot terminal. The saved
~4 MB work round trip (at 2^19) is dwarfed by the de-vectorization cost
(~400 ¬µs). Reverted; left a one-paragraph note at the `inverse_simd` odd branch
so nobody re-attempts it without the measurement.

### f32 wide: full fused terminal ‚Äî REVERTED (measured 10‚Äì16% slower)

Same idea for the paired-V4F path (`terminal3_inv_v4f_f32`, reading X, writing
paired work directly). Bit-identical output, but 10‚Äì16% slower. Here the f32
terminal crossing was *already* scalar, so nothing was de-vectorized ‚Äî the loss
came from **input locality**: the baseline terminal does a clean linear X sweep
(2 streams: `X[i]`, `X[n/2-i]`), while the fused terminal gathers 8 scattered
bins per butterfly (the split's `{i, 4M-i, 2M-i, 2M+i}` pattern applied to X).
The removed `pair_v2_to_v4f_f32` pass was cheap (cache-hot linear); the scattered
X gather it forced was not.

### f32 wide: fused `split4 + pair` ‚Äî KEPT (measured 5‚Äì7% faster)

The narrower fusion is the right one. Keep the linear terminal
(`terminal_radix2_inv_v_norm_f32`, cache-hot linear X sweep ‚Üí V2-f32 `output`),
then fuse **only** the first radix-4 split with the V2‚ÜíV4F pairing
(`split4_pair_v_norm_f32`): read the post-terminal slots from the cache-hot
`output`, and because split4-butterfly `i` already produces the two V2-slots
`{i, 2M+i}` that pack into paired-slot `i` and `{M+i, 3M+i}` that pack into
paired-slot `M+i` (P = n/8 = 2M), the split store and the pairing collapse to
in-register `CATLO`/`CATHI`. This deletes the standalone `pair_v2_to_v4f_f32`
pass (a full n-float read+write) and the split's in-place round trip, while
keeping the scattered access on cache-resident `output` rather than X.
Bit-identical to the three-pass baseline; **5‚Äì7% faster inverse at every odd
size** (512..524288 on NEON). `compute_vwork_inv_v4f` needed no change (it
already starts at n/8, below the fused n/2 split). The now-dead
`split4_v_norm_f32` and `pair_v2_to_v4f_f32` were removed.

### Lesson

The transport ("leaving the V4F manifold") the framing worried about was the
*cheap* part. What matters at these sizes is (a) not de-vectorizing the crossing
and (b) not turning a linear input sweep into a scattered gather. Fuse where the
scattered access is already cache-hot (split‚Üîpair), not across the input
boundary (terminal‚Üîsplit). The f64 path had no separate conversion pass to
remove, so it had nothing to gain and only vectorization to lose.

### Follow-up: why odd-size f32 *forward* is expensive (2026-07-03)

Question raised: DIT f32 forward looks disproportionately costly at N=8192.
Measured (NEON, ns/point): it is not 8192 specifically -- every odd log2(N)
forward sits ~15-20% above the even trend (2048/8192/32768), 8192 just stands
out between two cheaper evens.

Cause: the even f32 forward goes straight from the shared V4F chain into one
fused terminal (`terminal_v4f_f32`). The odd path instead ran, after the chain,
**three** extra full-array passes:
1. `depair_v4f_to_v2_f32` -- V4F->V2 layout conversion (n floats R+W),
2. `merge4_v_norm_f32` -- the extra radix-2-level merge (genuine work), and
3. a `for i: work[i] = terminal_work[i]` **full-array copy** (n R + n W, pure
   transport) -- present only because steps 1-2 left the merged V2 data in the
   `output` buffer, which the terminal must also write.
then the radix-2 terminal.

Two fixes, both bit-identical to the original, together **13-19% faster odd f32
forward** (8192: 8067 -> 6946 ns; 32768: 40.6 -> 33.0 us; 524288: 846 -> 719
us), and 8192's per-point cost is now in line with the even trend:
- **Buffer-role swap** removes the copy: run the shared V4F chain into the
  `output` buffer used as V4F scratch, depair into `work`, merge in place there,
  and let the terminal read `work` and write `output`. No copy, no aliasing.
- **`depair_merge4_v_norm_f32`** fuses the depair conversion into the merge
  (the forward mirror of the inverse `split4_pair`): paired-slot i depairs in
  registers to the four merge children {i, M+i, 2M+i, 3M+i}, so the standalone
  depair pass disappears. Source (V4F) and dest (V2) differ, so the in-place
  palindrome the standalone merge needed is dropped for a linear i sweep.

Removed the now-dead `depair_v4f_to_v2_f32`, `merge4_v_norm_f32`, and the
`butterfly4_v_norm_f32_v4` pointer wrapper. This time the fusion *won* (unlike
the inverse terminal3 attempts) for the same reason `split4_pair` won: the
scattered access stays on the cache-hot chain output, and no vectorized crossing
is de-vectorized -- the fused reads are contiguous paired-slot V4F loads.

## Cross-reference: the phase-packet FFT (2026-07-04)

The "third factorization" question (neither DIT nor DIF axis; diagonal walk
through phase space) was answered constructively. See
`notes/phase_packet_fft.md` + `experiments/phase_packet_fft.py`:

- The Stockham no-repack mechanism is a per-stage single-bit ledger move
  (verified); our real Bruun cells run under Stockham addressing unchanged,
  natural order both ends, width-agnostic vertical SIMD (the pairing-axis law
  of ¬ß1.2 reappears as the Stockham column tail ‚Äî same crossing depth).
- Gauging the tree by quantized time-shears (admissible iff sigma*q_t^2 = 0 mod
  N; freedom diamond 2^min(t,m-t)) yields exact phase-packet FFTs: 2-sparse
  stages, chirped-packet intermediates, butterflies pairing along diagonals,
  rotation constant along ONE (increment-)diagonal coordinate. DIT = sigma==0
  gauge point, and is cost-minimal: the family provably cannot reduce flops.
  It explains the Zak/torus "carry blocks" as inadmissible shear jumps.
