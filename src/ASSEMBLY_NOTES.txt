Assembly inspection notes, 2026-06-12

Build inspected
- Command used for the optimized object: make clean && make
- Command used for annotated assembly: g++ -Iinclude -O3 -std=c++17 -fPIC -Wall -Wextra -Wpedantic -mavx2 -mfma -S -fverbose-asm src/bfft.cpp -o build/src/bfft.s
- Command used for disassembly: objdump -drwC -Mintel build/src/bfft.o > build/src/bfft.objdump
- Host build selected the AVX2/FMA backend through the Makefile's automatic SIMD probe.

Things that looked healthy
- The core double and float normalization loops compile to VEX/AVX instructions and use FMA for the rotation expressions. I did not see legacy SSE instructions mixed into AVX hot loops; the 128-bit float tails are VEX-encoded XMM instructions, so they should not create an AVX/SSE transition penalty.
- The compiler emits vzeroupper on exits from AVX code paths. That is expected for ABI hygiene when returning to code that might use SSE.
- The f32 residue forward path is large, but the emitted loops still keep the wide arithmetic in registers and use unaligned vector loads/stores that match the library's public no-alignment contract.

Things that looked wrong or weird
- The generated norm2_inv_fused_f32 helper has very high register pressure. GCC spills several scalar twiddle arguments and a broadcast vector to the stack around its AVX2 and XMM-tail blocks. That is not ideal, but this helper is on the f32 inverse path; it is not part of the forward benchmark hot path, and changing it would be a larger scheduling/register-pressure rewrite rather than a safe small rescue.
- There are many vzeroupper instructions in the object because the header-only kernel inlines or emits several AVX-using API and helper bodies. That is not suspicious by itself, but it makes the disassembly noisy.
- The standard double-output policy previously switched to the sequential two-phase standard pack only for N > 8192. Benchmarks and the assembly shape suggest N == 8192 is already at the point where avoiding the fused native scatter plus later layout conversion can help. The policy boundary was the smallest safe tweak available from this inspection.

Small tweak made
- Changed the AVX2+ standard-output policy threshold from N > 8192 to N >= 8192. This keeps the existing automatic policy, does not add a user-facing compile flag, and only changes one boundary size.

Sydney notes, 2026-06-12

Build inspected
- Started by resetting the local main worktree to origin/main at 3f1ebef and removing untracked local files.
- Host is arm64 Apple clang, so the active Mac SIMD backend is neon-128. The Makefile did not add AVX flags, and the object is Mach-O arm64.
- Commands used:
  - make clean
  - make
  - c++ -Iinclude -O3 -std=c++17 -fPIC -Wall -Wextra -Wpedantic -S -fverbose-asm src/bfft.cpp -o build/src/bfft.s
  - make test

Assembly observations
- The main double and float kernels are vectorized on this machine. The generated assembly uses NEON vector loads/stores and arithmetic in the forward residue and stage bodies.
- The 128-bit double path expressed complex rotations as separate vmul plus vadd/vsub through V2_ADD(V2_MUL(...), V2_MUL(...)) and V2_SUB(V2_MUL(...), V2_MUL(...)). Apple clang did not reliably fuse those source forms in the forward hot path.
- The float32 path also has multiply/add rotation shapes, but quick benchmark passes showed enough f32 noise that changing it tripped the safety rule. I left f32 source arithmetic alone.
- The inverse path was also tried with fused helpers, but roundtrip timing became noisier at small sizes. I left inverse source arithmetic alone.

Small tweak kept
- Added V2_MADD and V2_MSUB to the 2-lane double abstraction. On SSE2 they expand to the old multiply plus add/subtract. On NEON they use vfmaq_f64 and vfmsq_f64.
- Replaced only the double 128-bit forward rotation sites, plus the residue-filter complex multiply, with V2_MADD/V2_MSUB. This keeps the x86 SSE behavior source-equivalent while letting the Mac NEON build emit fmla/fmls.

Benchmark notes
- Baseline and patched builds were compared against the same 11 default sizes from 512 through 524288.
- Two quick 20-iteration passes were noisy, so origin/main was also built in /private/tmp/bfft-baseline and compared with 100 iterations across those sizes.
- Focused high-iteration checks cleared the suspicious sizes:
  - N=512, 1000 iterations: baseline Std_ns 625.0, patched Std_ns 525.4.
  - N=1024, 1000 iterations: baseline Std_ns 1174.4, patched Std_ns 1047.0.
  - N=131072, 300 iterations: baseline Std_ns 439074.0, patched Std_ns 428148.1.
- No kept change showed a consistent >5 percent regression across 25 percent of tested sizes, and the focused repeats did not confirm any >20 percent single-size regression.

Sydney follow-up, 2026-06-12

Sync note
- The requested upstream sync landed on origin/main, not origin/master; this repository's remote HEAD is origin/main and no origin/master branch is advertised.
- Fast-forwarded local main from 5bbefc6 to 11af86b before this pass.

Architectural read
- The main locality win is still the fused depth-first residue architecture. The hot forward path keeps Bruun factor subtrees contiguous, uses the packed LeafTw/LeafTwF streams for sequential metadata reads, and avoids extra residue-domain store/reload passes by applying two tree levels at once before descending.
- The SSE/NEON abstraction was uneven: double 128-bit rotations already exposed multiply-add/subtract intent through V2_MADD and V2_MSUB, while the float 128-bit forward path used explicit multiply plus add/subtract expressions.
- Added V4F_MADD and V4F_MSUB. On SSE they expand to the old mulps plus addps/subps shape. On NEON they map to vfmaq_f32 and vfmsq_f32.
- Rewrote the forward float 128-bit rotation sites, including norm_q_fwd_f32, norm2_fused_f32, and codelet_d3_tw_res_f32, to use those agnostic primitives. Inverse f32 was left alone because this pass focused on the forward path and prior inverse experiments were timing-sensitive.

Assembly checks
- arm64 annotated assembly now shows fmla.4s/fmls.4s in the forward float segment and codelet regions.
- An x86_64 assembly-only compile with Apple clang succeeded. The generated SSE side stayed in mulps/addps/subps forms and did not require FMA instructions.
- The change does not alter memory traffic; it clarifies arithmetic intent and lets each backend choose the right primitive. Locality and cache behavior remain governed by the existing fused residue traversal and packed twiddle streams.

Locality follow-up, 2026-06-12

Measurement tool
- Added examples/locality_probe.cpp and wired it into Makefile and CMake example builds.
- The probe reports hot-cache timing against a rotating working-set timing. The ring uses independent input/work/output buffers sized by ring_mib, avoiding the subtraction artifacts from a separate eviction loop.

Converter locality
- native_to_standard_complex and native_to_standard_complex_f32 now iterate in standard bin order. This makes destination writes sequential and leaves the unavoidable permutation on the native read side.
- standard_to_native_complex and standard_to_native_complex_f32 now iterate in native position order. This makes native destination writes sequential.
- This follows the same architectural principle as the rest of the forward path: prefer linear writes and tolerate irregular reads when one side of a permutation must be nonlocal.

Locality measurements after the converter change
